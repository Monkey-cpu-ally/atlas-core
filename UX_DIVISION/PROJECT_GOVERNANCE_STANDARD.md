# ATLAS Project Governance Standard

**Version:** 1.0  
**Classification:** Executive Project Standard  
**Applies To:** Every ATLAS Division

## Purpose

This document establishes a measurable and transparent method for tracking progress across the ATLAS ecosystem.

ATLAS must never report estimated completion percentages as factual measurements.

All official progress must be calculated from documented deliverables stored in GitHub.

## Core Principle

If it cannot be measured, it cannot be reported as fact.

Every percentage shown in ATLAS documentation must be traceable to completed work.

## Definition of Complete

A deliverable is complete only when it has:

- A GitHub location
- Clear documentation
- Version number
- Review status
- Cross references where needed
- A defined owner or responsible division
- A checklist entry marked Complete

Draft work does not count as complete.

Ideas do not count as complete.

Conversation notes do not count as complete until committed to GitHub.

## Progress Formula

`Completed Deliverables / Total Planned Deliverables * 100`

Only deliverables marked **Complete** contribute toward official completion.

## Status Categories

- Not Started
- Planning
- Research
- In Development
- Testing
- Documentation
- Review
- Complete
- Archived

## Reporting Standard

Whenever ATLAS reports progress, it must distinguish between:

### Measured Progress

Calculated from completed deliverables.

### Estimated Progress

Professional judgment based on available information.

### Unknown Progress

Insufficient information to calculate.

These categories must never be mixed.

## Master Checklist Requirement

Every division must maintain a master checklist containing:

- Unique ID
- Deliverable name
- Description
- Dependencies
- Priority
- Current status
- Completion date
- Version

## Quality Standard

A document is not complete simply because it exists.

It must be understandable, organized, consistent with ATLAS standards, and placed correctly in GitHub.

## Audit Process

Periodic audits should check:

- Folder organization
- Missing deliverables
- Duplicate work
- Broken references
- Version consistency
- Checklist accuracy
- Outdated plans

## Final Principle

Trust is built through honesty.

ATLAS will not present guesses as facts.

When a number is official, the user must be able to trace it back to a checklist, deliverable, and GitHub location.
